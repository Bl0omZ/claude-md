# CLAUDE.md 生成器

产出简短、可执行、路由风格的 CLAUDE.md 文件。分为项目级和用户级两种模式。

## 运行原则

以下原则约束本 skill 的行为，也约束其产出内容：

- **先想再写。** 说出你推断出的结论。如果多种结构都合适，列出选项——不要自己悄悄选。拿不准就问。
- **简单优先。** 用最精简的 CLAUDE.md 解决问题。不要写推测性内容。
- **审计时精准修改。** 只改出问题的部分。保持用户原有的语气。
- **目标驱动。** 每个输出必须以一个可验证的检查收尾。

## 第 0 步：选择范围

每次必须先问清楚：

- **项目级** → `<仓库>/CLAUDE.md` 或 `<仓库>/<模块>/CLAUDE.md`。描述本代码库。
- **用户级** → `~/.claude/CLAUDE.md`。描述用户本身。对所有项目生效。

**兜底：** 如果用户未指定且 agent 也没问，默认按项目级处理。

后续流程以此分支。

## 项目级工作流

### 硬约束

1. **≤ 200 行。** 超出部分 → 放到 `docs/*.md`，绝不内联。
2. **每条规则必须能通过 5 秒测试。**
3. **必须有 `不要引入` 块。** 不少于 3 条，每条附带一行原因。
4. **不要写散文、历史、宣传语、价值观。**
5. **敏感目录（`auth/`、`billing/`、`infra/`、`migrations/`）必须有独立的 CLAUDE.md。**
6. **全文语言统一。** 不要在文件中混用语言。根据用户的语言或已有 CLAUDE.md 来决定。技术术语（库名、CLI 命令、文件路径）不算混用。

### 生成前扫描

先检查，再提问：
- 清单文件：`package.json` / `pyproject.toml` / `Cargo.toml` / `go.mod`
- 顶级目录 → 标记需要独立 CLAUDE.md 的敏感模块
- 已有的 `CLAUDE.md` / `MEMORY.md` / `docs/` / `.claude/hooks/`

说出你发现了什么。只问那些无法推断的内容（优先级排序、工作风格调整、`不要引入` 的补充项）。

### 骨架

```markdown
# <项目名>

## 概述
<一行：做什么 + 面向谁>
<一行：优先级排序，如 "速度 > 功能 > 打磨">

## 技术栈
- <库>@<主版本号>

## 不要引入（除非明确要求）
- <库> —— <一行原因>

## 编码规则
- <祈使句，可验证的规则>

## 工作风格
- 显式声明假设；不确定就问，不要默选
- 遇到模糊需求，列出不同解读——不要自己悄悄选
- 用最小改动解决问题；不要做推测性抽象
- 匹配现有代码风格，即使你自己不这么写
- 多步任务先列计划，附带验证步骤
- 不要写"问得好！"/"很高兴帮你！"之类的废话

## 指针（按需阅读）
- 架构：`docs/architecture.md`
- ADR：`docs/adrs/`

## Hooks（强制执行，不靠记忆）
定义在 `.claude/hooks/`。列在此处以备查看：
- <hook>

## Memory
每个任务开始前读取 `MEMORY.md`。完成后追加发现。
```

工作风格默认项是不可协商的起点。如果想移除某一项，须说明移除了什么以及为什么。

## 用户级工作流

用户级 CLAUDE.md 关注**用户是谁**和 **Claude 的默认行为**。输入方式：**访谈**，而非扫描。

### 访谈

分 3~4 轮进行，每轮集中问几个问题。不必每个问题都问，能推断的就跳过。根据前面的回答灵活调整。总共控制在 12 个问题以内。

**第 1 轮 — 角色与领域**
- 主要做什么方向？（后端 / 前端 / 全栈 / ML / 基础设施 / 移动端 / 混合）
- 未指明语言时的默认选择？

**第 2 轮 — 技术栈默认值**
- 首选框架 / 工具？
- 哪些东西你绝对不用？（库、模式、范式）

**第 3 轮 — 工作风格**
- 遇到模糊需求：倾向于问清楚还是做最佳猜测？
- 计划风格：提前规划还是按需规划？
- 代码注释：简洁就好还是详细文档？

**第 4 轮 — 沟通**
- 回复用什么语言？代码注释用什么语言？
- 语气 / 正式程度？
- 雷区？哪些写法让你反感？（客套话、emoji、过度道歉、模棱两可）

### 骨架

```markdown
# 个人默认设置

## 关于我
- 角色：<如 "后端工程师；侧重基础设施与 DX">
- 主要领域：<列表>

## 默认技术栈
- 未指明时的语言优先级：<如 "Go > Python > TS">
- 默认框架：<如 "Python Web 用 FastAPI，前端用 Next.js">
- 全局禁用：<列表，每条附带一行原因>

## 工作风格
[阅读 `references/karpathy-claude.md` 并根据访谈回答进行适配。
保留 4 条原则：先想再写 / 简单优先 / 精准修改 / 目标驱动。
用自己的话表达，但不要悄悄丢掉原则。]

## 沟通
- 回复语言：<如 "中文">
- 代码注释：<如 "英文">
- 语气：<一行>
- 不要：<雷区列表>

## 我的取舍偏好
- <如 "可读性 > 聪明">
- <如 "无聊的技术 > 新颖的技术">
- <如 "显式 > 隐式">
```

### 硬约束（用户级）

1. **≤ 150 行。** 比项目级更严格——因为它对所有项目生效。
2. **不得包含特定项目的内容。** 如果某条规则只适用于一个仓库，它应该放在那个仓库的 CLAUDE.md 里。
3. **每条偏好必须可执行。** "我喜欢干净的代码" → 不合格。"偏好 named export" → 合格。
4. **全文语言统一。** 与项目级同一规则，不要混写。

## 规则改写（两种模式通用）

| ❌ 模糊 | ✅ 可验证 |
|---|---|
| 写干净的代码 | 使用 named export，路由文件除外 |
| 类型安全 | 不使用 `any`；用泛型或接口 |
| 组件保持小巧 | 组件 ≤ 200 行，除非有充分理由 |
| 现代异步写法 | `async/await`，不用 `.then()` 链 |
| 命名好一点 | 用完整单词，`id`、`url`、`ctx` 除外 |

## 审计已有文件（精准模式，两种通用）

1. 统计行数。项目级 >200 / 用户级 >150 → 标记需要精简的段落。
2. 标记未通过 5 秒测试的规则，给出针对性改写建议。
3. 标记应放到别处的内容（项目级放 `docs/`；用户级中的项目特定规则应移除）。
4. 检查是否缺少 `不要引入` 块（项目级）或 `不要` 列表（用户级）。
5. 先提删除，再提交新增。保留原有语气和段落顺序。

## 输出检查（两种通用）

输出行数 + 30 秒测试：
- 项目级：一个陌生人能否回答"这是什么项目 / 技术栈 / 新代码往哪放"？
- 用户级：一个陌生人能否回答"这个人做什么方向 / 默认选什么技术 / 怎么沟通"？

如果无法通过，先修好再声明完成。

## 没内容就删掉，不要占位

如果骨架中某个段落对当前项目/用户没有实际内容，**直接删掉整个段落**。绝不写占位条目，例如：
- "Hooks：无" / "Memory：N/A" / "未配置 hooks"
- "非 Python 项目" / "无框架" / "不适用"
- 任何值为"无"、"none"、"N/A"、"——"或语义等价的条目

一个只保留相关段落的短文件，永远好过一个用空条目填满的"完整"文件。

## 拒绝添加的内容

- 项目级："项目历史" / "使命" / "价值观" 类段落
- 用户级：个人宣言、哲学随笔、生平细节
- 二者通用：内联的架构图或 ADR 内容；以"尽量 / 考虑 / 最好 / 理想情况"开头的规则；没有具体示例的语气指导
